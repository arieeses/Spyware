# 内鬼系统 · 立项建议书

> 面向 v2board 机场的「内鬼/二次分销」检测与风控中央控制面板

---

## 1. 项目定位

给运营的 v2board 机场做一套**中央风控面板**，核心目标：识别把订阅二次分销、多人共享、白嫖倒卖的「内鬼」用户，量化风险后自动/半自动处置，并支持**多前端面板统一管控**。

一句话：**用行为数据给每个用户打「内鬼分」，高分的隔离、投毒或封禁，且不影响正常用户。**

关键判断：这是一个**概率性检测**问题，不存在 100% 准确的规则。设计上必须从第一天就考虑**误杀成本**——把「直接封」和「先隔离观察」分开，给正常用户留申诉/自动恢复通道。这是本项目最容易翻车的点，放在最前面强调。

---

## 2. 核心检测信号（内鬼画像）

按数据来源和可信度分层，建议 MVP 先做 ★★★ 的：

| 信号 | 说明 | 数据来源 | 权重 |
|------|------|---------|------|
| **订阅拉取 IP 数** | 同账号短时间大量不同 IP / 不同 ASN / 不同地区，是最强信号。正常用户 3~10 个 IP，分销面板聚合拉取会有几十上百 | Nginx access log（订阅路由） | ★★★ |
| **订阅拉取频率与节奏** | 二次分销面板通常**定时批量拉取**，节奏规整（如每 5min 整点）；正常客户端拉取零散 | Nginx log | ★★★ |
| **拉取 User-Agent 分布** | 同账号出现多种客户端 UA、或出现分销面板特征 UA（v2board/sub-store/机场助手等） | Nginx log | ★★★ |
| **在线设备/并发数** | 同一时刻多节点多 IP 并发连接，远超套餐设备数 | 节点上报 / v2board online_count | ★★ |
| **流量形态** | 流量消耗与套餐/时段不匹配、7×24 满速跑 | v2_stat_user | ★★ |
| **节点接口访问行为** | 请求节点列表接口的模式、重放订阅 token 的来源分散度 | Nginx log / v2board | ★★ |
| **注册与支付特征** | 批量注册、同支付源、邀请链裂变异常 | v2board DB | ★ |

> IP 维度建议引入 **ASN + IP 库（GeoIP/纯真）**，用「不同 ASN 数」和「跨国跨省跳变」比单纯 IP 计数更抗干扰（家宽拨号会换 IP，但 ASN 不变）。

---

## 3. 评分模型（正常 / 低 / 中 / 高）

**分阶段做，别一上来就上机器学习。**

- **阶段一（MVP）：规则加权评分。** 每个信号归一化到 0~100，按权重加权求和，落到四档阈值：
  - `正常（0~30）` / `低（30~50）` / `中（50~75）` / `高（75~100）`
- **阶段二：滑动窗口 + 基线。** 用户自身的历史行为做基线，检测「突变」（某天 IP 数从 5 跳到 80），比绝对阈值更准，也更抗误杀。
- **阶段三（可选）：无监督聚类 / 异常检测**（如 Isolation Forest），把明显偏离群体的账号挑出来复核，作为规则的补充而非替代。

设计要点：
- **分数可解释**：面板上要能点开看「为什么这个用户是高风险」——列出命中的信号和贡献值。黑盒分数运营不敢用。
- **分数带时间衰减**：老的违规逐渐降权，避免用户永久背锅。
- **人工标注回流**：运营确认的「真内鬼」和「误杀」要能回写，用于调阈值和后续训练。

## 4. 处置策略（分级、可逆优先）

按风险档位配置不同动作，**默认动作要可逆**：

| 档位 | 建议动作 |
|------|---------|
| 低 | 仅记录、观察，进观察名单 |
| 中 | 打标 + 告警；可选**限速/限并发**；订阅拉取加频率限制 |
| 高 | **单独风险分组** + **独立订阅域名**（隔离，随时可封不影响正常用户）；可选**订阅投毒**（返回降速/假节点/带水印节点以定位分销面板）、**降速**、封号 |

独立域名/分组的价值（你提到的核心玩法）：
- **隔离爆破半径**：高风险用户全部走独立订阅域名，一旦确认可整域封禁或整组降速，正常用户零影响。
- **蜜罐定位**：给不同高风险用户下发**带唯一水印的节点**，反查是哪个源头在分销（谁的订阅被贴到了公开面板/TG 群）。
- **梯度处置**：先降速观察，用户若是误杀会申诉，真内鬼一般沉默——用行为反证降低误杀。

> 建议所有处置都走**审计日志**：谁、什么时间、因为什么分数、做了什么动作、是否可回滚。

## 5. 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                   中央控制面板 (Web UI)                     │
│   用户风险列表 / 画像详情 / 处置操作 / 多面板切换 / 审计       │
└───────────────┬─────────────────────────┬────────────────┘
                │ API                     │ API
┌───────────────▼──────────┐   ┌──────────▼────────────────┐
│      评分/分析引擎          │   │       处置执行器            │
│  特征提取→评分→分档→告警    │   │  分组/换域名/限速/封禁/投毒   │
└───────────────▲──────────┘   └──────────▲────────────────┘
                │                          │
┌───────────────┴──────────────────────────┴────────────────┐
│                     数据采集层 (Collector)                  │
│  ① Nginx access log（1Panel/aaPanel，订阅+节点接口）         │
│  ② v2board DB（user / stat / server / online）             │
│  ③ 节点上报（可选，在线数/并发）                             │
└────────────────────────────────────────────────────────────┘
        ▲ 每个机场部署一个 Agent，上报到中央面板
   [机场A] [机场B] [机场C] ...  ← 多面板统一纳管
```

**多面板对接方案**：中央面板做「主控」，每个 v2board 实例旁部署一个轻量 **Agent**（读日志 + 读 DB + 执行处置），通过统一 API 与主控通信。主控只存聚合后的风险数据和指令，**不直连各机场数据库**——解耦、安全、易扩展。

---

## 6. 关键对接点

### 6.1 v2board
- **读**：`v2_user`（订阅 token/uuid、套餐、状态）、`v2_stat_user`（流量）、在线数、订阅访问记录。
- **写/处置**：改用户分组（`group_id`）、封禁（`banned`）、限速。优先通过 **v2board 后台 API** 操作，避免直接写库导致缓存不一致；确需写库时走事务 + 触发其缓存刷新。
- **订阅域名隔离**：给风险组配置独立 subscribe URL 域名/路由（Nginx 层按 group 或 token 分流），这是「独立域名」落地的关键。

### 6.2 1Panel / aaPanel（日志源）
- 主要吃 **Nginx access log**：订阅路由（`/api/v1/client/subscribe?token=xxx`）和节点接口的访问记录，含 IP、时间、UA、path。
- 采集方式二选一：
  - **实时**：Agent `tail -F` 日志 → 解析 → 增量上报（推荐，延迟低）。
  - **批量**：定时读取 1Panel/aaPanel 日志目录/接口拉全量。
- 注意 **CDN/反代场景**：日志里的 IP 可能是 CDN 回源 IP，需正确解析 `X-Forwarded-For` / `X-Real-IP`，否则 IP 信号全废。**这是数据质量的头号坑，立项时就要确认反代链路。**

### 6.3 多前端面板批量控制
- 统一 Agent 协议（HTTP + 鉴权 token），主控下发指令、Agent 执行并回执。
- 支持**批量操作**：一键对全网某档位用户降速/换域名/封禁。

## 7. 技术选型建议

| 层 | 建议 | 理由 |
|----|------|------|
| Agent | Go 单二进制 | 好部署、并发读日志强、无依赖，适合塞进各机场 |
| 主控后端 | Go / Python(FastAPI) | 团队熟悉优先；Python 便于后续接分析/ML |
| 存储 | PostgreSQL + Redis；日志明细可选 ClickHouse | 明细量大用列存，聚合快；Redis 做频率计数/滑窗 |
| 前端面板 | Vue/React + 现成 admin 模板 | 快速出可用 UI |
| IP 库 | MaxMind GeoLite2 / 纯真 + ASN | ASN 维度判分销更准 |

> 若追求快速验证，MVP 阶段可先不上 ClickHouse，PostgreSQL + 定时聚合就够。

---

## 8. 分期里程碑

- **M0 · 数据打通（1~2 周）**：单机场 Agent 跑通，解析 Nginx 订阅日志 + 读 v2board DB，正确拿到 IP/UA/频率。**先验证反代 IP 解析正确**。
- **M1 · MVP 评分面板（2~3 周）**：规则加权评分 → 四档 → 用户风险列表 + 画像详情（可解释）。只读，不处置。人工看准不准。
- **M2 · 处置能力（2 周）**：风险分组、独立订阅域名、限速/封禁，全部带审计和回滚。
- **M3 · 多面板纳管（2 周）**：多 Agent、统一主控、批量操作。
- **M4 · 增强（持续）**：基线突变检测、订阅投毒/水印溯源、异常检测模型、误杀申诉回流。

---

## 9. 风险与注意点

1. **误杀是第一风险**。正常重度用户（多设备 + 家庭共享 + 频繁切节点）会被误判。→ 处置分级 + 可逆 + 观察期 + 申诉回流，别一刀切封号。
2. **IP 解析正确性**决定整个系统成败（CDN/反代/IPv6/家宽换 IP）。→ M0 必须先啃下来。
3. **对抗升级**：内鬼发现被抓会改行为（降频拉取、分散 IP）。→ 信号要多维交叉，别只靠单一 IP 计数；模型要能迭代。
4. **性能**：大机场日志量大。→ Agent 侧做预聚合，别把明细全上传主控。
5. **合规/边界**：本系统用于**运营方对自有服务的反滥用**（防二次分销、防白嫖），属正当风控。涉及用户 IP 等数据，注意存储加密、访问审计、保留期限。
6. **v2board 版本差异**：不同版本/魔改版 DB schema 和订阅路由不同。→ Agent 做适配层，schema 可配置。

---

## 10. 给决策者的一句话建议

**先做 M0+M1（只检测不处置）**，用真实数据验证「打分准不准、误杀多不多」——这个投入小、风险低，但直接决定项目是否成立。评分跑准了再做处置和多面板，避免一上来就重投入却栽在数据质量和误杀上。
