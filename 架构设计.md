# 内鬼系统 · 架构设计（优化定稿）

> 配套:[内鬼识别方案.md](内鬼识别方案.md)（信号与逻辑）· 本文定架构与落地路线

---

## 1. 关键取舍（优化决策）

| 决策点 | 选择 | 理由 |
|--------|------|------|
| MVP 是否用 Agent | **否,单服务起步** | 契合"运维极简";直连 v2board(只读)+ 推送日志即可。Agent 留到多机场扩展期 |
| 开发语言 | **Python** | 日志解析/评分/后续 ML 一条龙,单语言好维护 |
| 日志接入 | **推送 >** 拉取 | 面板 API 扛不了实时日志;Nginx syslog / vector 推送 |
| 信号落地顺序 | **订阅层优先** | ASN/UA/频率/流量背离/注册轨迹先做;节点侧(IP静默/扫描/TLS)留接口 |
| 评分 | **规则加权 + 可解释** | 每信号输出贡献值+理由,先别上黑盒 ML |
| 第一增量依赖 | **零外部依赖(纯标准库)** | clone 即跑,先验证准确性 |

## 2. 系统分层

```
┌───────────── 中央控制面板(主控 · Python) ─────────────┐
│  Web UI(风险名单/画像/处置) · 控制通道 · 审计         │
├───────────────────────────────────────────────────────┤
│  分析引擎:排除层 → 多维评分 → 跨token关联 → 名单       │
├───────────────────────────────────────────────────────┤
│  数据接入(connector):                                 │
│   · 订阅拉取日志(Nginx neigui 格式, 推送)             │
│   · v2board(MySQL 只读: token/套餐/注册时间/流量)      │
│   · 节点侧连接日志(按IP, 喂养 IP静默/扫描/使用形态)    │
│   · ASN 分类(MVP: CIDR名单 → 生产: GeoLite2-ASN)      │
└───────────────────────────────────────────────────────┘
        ▲ 扩展期: 每前端旁路 Agent(执行处置 + 推日志)
```

## 3. 信号落地矩阵

| 信号 | 数据源 | MVP | 说明 |
|------|--------|-----|------|
| ASN(机房/住宅) | 拉取日志+ASN库 | ✅ | 生产替换 GeoLite2 |
| UA(客户端/工具) | 拉取日志 | ✅ | 白名单+工具正则 |
| UA伪造(客户端UA+机房) | 上两者 | ✅ | 强信号 |
| 拉取频率(机器规整) | 拉取日志 | ✅ | 间隔变异系数 |
| 流量背离 | v2board 流量 | ✅ | 抗伪装 |
| 注册→拉取轨迹 | v2board+日志 | ✅ | 新号判别 |
| 拉取后IP静默 | 节点侧连接 | ⏳ | 需节点日志 |
| 扫描式短连 | 节点侧连接 | ⏳ | 需节点日志 |
| TLS/HTTP指纹 | Nginx JA3 | ⏳ | 需JA3模块 |
| 跨token关联 | 全量 | ⏳ | 提召回 |

## 4. 落地路线（增量交付）

- **增量 1 · 检测核心(本次开写)**:日志解析 + ASN/UA 分类 + 排除层 + 规则评分 + 风险名单 CLI。零依赖、可跑、验证准确性。
- **增量 2 · 数据接入**:v2board MySQL 只读 connector(token→画像自动补全)+ 日志推送落库。
- **增量 3 · Web 面板**:风险名单/画像详情/证据链展示(FastAPI + 简单前端)。
- **增量 4 · 处置与控制通道**:高危组分配、独立订阅域名、批量管控(幂等+回执+审计)。
- **增量 5 · 节点侧信号 + 指纹**:接节点连接日志(IP静默/扫描),Nginx JA3(TLS指纹)。
- **增量 6 · 增强**:跨token关联、无监督聚类、跨机场情报共享、Agent 化。
